The finale earns its position: one system that makes your game work on any device,
and one pattern that makes it infinitely extensible. Then everything connects.

## The new Input System — devices become irrelevant

Open **`Module10_Input/Scenes/01_InputSystem`**. The old way,
`Input.GetKey(KeyCode.W)`, hard-codes the keyboard. The new way inverts it:

1. Create an **Input Actions** asset → Action Map "Gameplay" → Actions: `Move`
   (Value/Vector2), `Jump` (Button), `Interact` (Button).
2. Bind `Move` to BOTH the WASD composite and the gamepad left stick.
3. Tick **Generate C# Class** — Unity writes the typed wrapper for you.

Your code reads the *action*, never the device:

```csharp
Vector2 input = _actions.Gameplay.Move.ReadValue<Vector2>();
```

Plug in a controller mid-game: it just works, zero code changes. `InputHandler.cs`
also shows the event style — and its golden rule, an old friend from Coin Rush's
events chapter:

```csharp
private void OnEnable()  => _actions.Gameplay.Jump.performed += OnJump;
private void OnDisable() => _actions.Gameplay.Jump.performed -= OnJump;
```

Every `+=` has its `-=`. Mobile bonus: **On-Screen Stick** components virtually press
gamepad bindings — touch support with zero new input code.

## IInteractable — the most important pattern in the course

Open **`02_Interaction`** and read the shortest file in the module:

```csharp
public interface IInteractable
{
    void Interact(GameObject interactor);
    string GetPromptText();
}
```

An **interface** is a contract with no body. `InteractionSystem.cs` raycasts from the
camera, and if the hit object implements `IInteractable`, shows its prompt and calls
`Interact()` on **E**. Here's the part to sit with: the player script imports neither
`Door.cs` nor `Chest.cs` — it has *no idea what it's interacting with*. The door
swings, the chest spawns loot, and the player code is identical for both.

Add an NPC, a lever, a terminal next month: implement the interface, done — the
interaction system is *never edited again*. Systems that grow without being modified:
you met this idea as polymorphism in OOP Pillars; here it wears its industry name,
**programming to an interface**, and it powers every RPG interaction prompt you've
ever seen.

## Interview corner

**Biggest advantage of the new Input System?** Actions abstract devices — bind
keyboard, gamepad, and touch to one action; code reads the action, so new devices
cost zero code.

**How does IInteractable enable extensibility?** The interaction system depends only
on the contract; new interactable types implement it without touching existing code —
the Open/Closed Principle in practice.
