Open **Packages → Ember Hollow → Scripts → Environment → `PostFXController.cs`**,
`CollapseTimer.cs`, and `GameWiring.cs`.

## The problem — the whole screen needs to feel the collapse

As the countdown nears zero, the game should feel more oppressive — vignette
closing in, colors desaturating — without hand-darkening every individual light
and material in the scene.

## The fix — driving a Volume profile at runtime

```csharp
private void HandleProgress(float t)
{
    if (_vignette != null)
        _vignette.intensity.value = Mathf.Lerp(vignetteStart, vignetteEnd, t);

    if (_colorAdjustments != null)
        _colorAdjustments.saturation.value = Mathf.Lerp(saturationStart, saturationEnd, t);
}
```

`.value`, not a plain field assignment — these are `VolumeParameter<T>` wrappers,
and `TryGet` (used in `Awake()`) returns `false` instead of throwing if the profile
doesn't have that override added, so always check before using one:

```csharp
volume.profile.TryGet(out _vignette);
volume.profile.TryGet(out _colorAdjustments);
```

## Where `t` comes from — CollapseTimer owns the truth

```csharp
public event Action<float> OnProgressChanged;
...
_elapsed += Time.deltaTime;
float curved = config.intensityOverTime.Evaluate(Progress01);
OnProgressChanged?.Invoke(curved);
```

`RuinConfig.intensityOverTime` is an `AnimationCurve` — an instructor can reshape
*when* the intensity ramps up (slow burn vs. sudden dread) entirely in the
Inspector, with zero code changes.

## The whole series, wired in one file

Every system you've studied — the golem, the timer, the mixer, the post-fx — could
each hold direct references to the others. Instead, exactly one file does that job:

```csharp
private void OnEnable()
{
    if (golem != null && audioMixerManager != null)
        golem.OnRoar += audioMixerManager.DuckMusicForRoar;

    if (collapseTimer != null && audioMixerManager != null)
        collapseTimer.OnProgressChanged += audioMixerManager.SetIntensity;
}
```

<EventFanout source="CollapseTimer.OnProgressChanged" hub="GameWiring" listeners="AudioMixerManager.SetIntensity,PostFXController.HandleProgress" note="GolemAI.OnRoar also fans out here, to DuckMusicForRoar" />

Delete `GameWiring.cs` entirely and every other script in this package still
compiles and runs standalone — the golem still chases, the timer still counts
down, the walls still dissolve on schedule. Only the *cross-system reactions* stop:
music won't duck for roars, the mix won't shift with the timer. That's the proof
this is decoupling, not just indirection for its own sake.

## The counterfactual

Give `GolemAI` a direct `[SerializeField] AudioMixerManager` reference and call
`DuckMusicForRoar()` straight from `TickChase()`. It works — until the next system
wants to react to a roar too (a screen shake, an achievement), and now `GolemAI`
needs a reference to that as well. `GameWiring` is the one place allowed to know
about all of them.
