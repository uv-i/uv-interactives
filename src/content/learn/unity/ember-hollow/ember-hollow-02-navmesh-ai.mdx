## The problem — chasing around corners, not through them

The golem needs to path around rubble and through corridors toward you, not walk
straight at your position and get stuck on the first wall. Doing this by hand —
raycasting around obstacles, backtracking on dead ends — is a real, hard
pathfinding problem you do not want to write from scratch.

## The fix — NavMeshAgent + an explicit state enum

```csharp
public enum GolemState { Patrol, Chase }

[RequireComponent(typeof(NavMeshAgent))]
public class GolemAI : MonoBehaviour
{
    private NavMeshAgent _agent;
    private GolemState _state = GolemState.Patrol;
```

`NavMeshAgent` does the actual pathfinding — our job is only deciding *when* to
chase, never *how* to get there:

```csharp
switch (_state)
{
    case GolemState.Patrol:
        TickPatrol(distToPlayer);
        break;
    case GolemState.Chase:
        TickChase(distToPlayer);
        break;
}
```

Each state just re-targets the agent's destination:

```csharp
private void TickChase(float distToPlayer)
{
    _agent.speed = config.chaseSpeed;
    _agent.SetDestination(player.position);
    ...
```

## The counterfactual

Replace the `switch` with the bool-flag version — `isChasing`, `wasChasing`,
`isPatrolling` — and adding a third state (say, `Stunned`) means auditing every
existing `if` for a case you forgot. The enum + switch keeps "what am I doing right
now" as one readable answer instead of a web of flag combinations.

## Try it — break it on purpose

In `GolemConfig`, set `sightRange` higher than `loseInterestRange`. Play. The golem
will chase forever once it notices you, because `TickChase`'s exit condition
(`distToPlayer > loseInterestRange`) can never fire before `TickPatrol` would have
re-triggered chase anyway. Fix the values back and explain in one sentence why the
ordering of those two numbers matters.

## Challenge

Add a third state, `Stunned`, entered when the player uses some future ability, that
sets `_agent.isStopped = true` for 2 seconds before returning to whichever state
it was in before.

## Interview corner

**Why a NavMeshAgent instead of moving the transform toward the player directly?**
NavMeshAgent paths around static geometry baked into the NavMesh; a direct
`Vector3.MoveTowards` toward the player walks straight into any wall between them.

**What's the difference between `_agent.SetDestination()` and setting
`transform.position` directly?** `SetDestination` asks the agent to *path* there over
time, respecting the NavMesh and avoiding obstacles; setting `transform.position`
teleports instantly and bypasses the agent's pathing (and desyncs the agent's
internal state from the object's actual position).
