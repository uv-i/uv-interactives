Open `PlayerAnimator.controller` in the Project window (built by
**Ember Hollow → Step 1**), then look at how it's built in
**Packages → Ember Hollow → Scripts → Editor → `SceneBootstrapper.cs`**.

## The problem — no popping between clips

Idle, Walk, and Run are three separate animation clips. Hard-switching between them
based on `if (speed > 2f) PlayWalk()` produces a visible pop the instant speed
crosses a threshold — the pose jumps instead of transitioning.

## The fix — a 1D blend tree

`SceneBootstrapper.cs` builds the whole controller through the public
`UnityEditor.Animations` scripting API, so Step 1 hands you a working, inspectable
controller instead of an empty one:

```csharp
var tree = new BlendTree { name = "Locomotion" };
AssetDatabase.AddObjectToAsset(tree, controller); // BlendTrees must be sub-assets
tree.blendType = BlendTreeType.Simple1D;
tree.blendParameter = "Speed";
tree.AddChild(idle, 0f);
tree.AddChild(walk, 2.2f);
tree.AddChild(run, 5.5f);
```

<TreeDiagram root="Locomotion blend tree|Simple1D on Speed" leaves="Idle|0,Walk|2.2,Run|5.5" note="thresholds must ascend for a 1D blend to interpolate correctly" />

Every frame, `PlayerController.cs` feeds one number into that tree:

```csharp
if (animator != null)
    animator.SetFloat(speedParam, targetSpeed);
```

No `PlayWalk()`/`PlayRun()` anywhere — the tree interpolates continuously between
whichever two clips bracket the current `Speed` value.

## The counterfactual

Remove the blend tree and wire three `Animator.Play("Idle"/"Walk"/"Run")` calls
triggered by speed thresholds instead. Walk at exactly the threshold speed and the
animation will visibly snap every time speed crosses it — no interpolation, no smoothing.

## Try it — break it on purpose

Open the Blend Tree Inspector and swap the `Walk` and `Run` thresholds (2.2 and 5.5)
so they're out of order. Play and scrub the Speed parameter in the Animator's live
preview. The blend goes wrong — a 1D blend tree requires ascending thresholds to
interpolate sanely. Put them back once you've seen it break.

## Interview corner

**What problem does a blend tree solve that a plain state machine doesn't?**
A state machine transitions discretely between states (with a transition duration
at best); a 1D blend tree interpolates continuously across a whole range of a
parameter, so intermediate speeds get intermediate blends of the surrounding clips
instead of a snap.

**Why must a BlendTree be added via `AssetDatabase.AddObjectToAsset` instead of just
assigning it?** A `BlendTree` is a sub-asset — it has to live inside the
`.controller` asset file on disk to be saved and referenced correctly; creating it
in memory without attaching it to the controller asset would lose it the moment
the editor session ends.
