Three chapters of setup pay off here. Open
**Packages → OOP Pillars → Scripts → `ScreenManager.cs`** — the shortest important
file in the package.

## The list that shouldn't work

```csharp
[SerializeField] private List<BaseScreen> _allScreens = new List<BaseScreen>();
```

One list. Typed `BaseScreen` — a class that (Chapter 2) **can never be instantiated**.
So what's actually in there? A SplashScreen, a LoadingScreen, a GameplayHUD, a
GameEndScreen — and your PauseScreen from Chapter 3. Five different classes in a
container that claims to hold one type.

That's the first half of polymorphism: **a base-class reference can hold any subclass**.
Check the Inspector: you dragged five differently-named GameObjects in, and Unity shows
them all as BaseScreen. Polymorphism isn't just in code — it's in the editor.

## The dispatch — where it gets good

```csharp
public void ShowScreen(int index)
{
    if (_currentScreen != null)
        _currentScreen.Hide();

    _currentScreen = _allScreens[index];
    _currentScreen.Show();          // ← the magic line
}
```

`_currentScreen` is typed `BaseScreen`. Calling `.Show()` runs BaseScreen's `Show()` —
which calls `OnShow()` — and *that* dispatches to whichever override the actual object
carries: the splash starts its timer, the HUD resets its score, your pause screen
freezes time. **The caller never changes. Only the result does.**

Prove it: each screen's `OnShow()` already has a `Debug.Log` with its class name.
Play, watch the Console — one line of calling code, four different log messages.

## The counterfactual — read it and wince

Without polymorphism, `ShowScreen` looks like this, forever:

```csharp
if (screen is SplashScreen s)      s.Show();
else if (screen is LoadingScreen l) l.Show();
else if (screen is GameplayHUD h)   h.Show();
else if (screen is GameEndScreen g) g.Show();
// add PauseScreen? edit this. add SettingsScreen? edit this. forever.
```

Every new screen means reopening this file. With polymorphism, `HideAll()` is:

```csharp
foreach (BaseScreen screen in _allScreens)
    screen.Hide();
```

Your PauseScreen was hidden by that loop the moment you added it to the list —
**ScreenManager's code hasn't changed since before your screen existed.** Systems
that grow without being edited: that's the closed-for-modification, open-for-extension
idea you'll later meet formally as the Open/Closed Principle.

## The four pillars, one sentence each

- **Encapsulation** (Ch1) — data is private; one door to change it.
- **Abstraction** (Ch2) — the contract says *what*, never *how*.
- **Inheritance** (Ch3) — subclasses get the foundation free, add only what's new.
- **Polymorphism** (Ch4) — one call on the contract, each subclass answers in its own voice.

Notice they stack: polymorphism needs inheritance, inheritance needs the abstract
contract, and the contract stays trustworthy because encapsulation guards the state.
It was never four separate tricks — it's one design, seen from four sides.
